# Examples

## Зачем

Здесь должен лежать **пример проекта**, на котором мы демонстрируем работу скилла `openspec-task-explain`.

Пример готовится **до запуска GigaCode CLI** и не входит в one-shot реализацию. Это подготовительный артефакт.

## Что должно быть в `input/`

Мини Python-проект с openspec-структурой:

```
input/
├── openspec/
│   └── changes/
│       └── add-payment-validation/    ← имя change на твой выбор
│           ├── change.md              ← заполнен по шаблону openspec-propose
│           ├── design.md              ← заполнен по шаблону openspec-design (16 разделов)
│           └── tasks.md               ← с задачами в формате [ ] N.M
├── src/
│   └── (несколько Python-файлов, упомянутых в design.md)
└── tests/
    └── (несколько pytest-файлов)
```

**Важно:** код в `src/` и `tests/` должен быть **минимальным и работающим** (хотя бы импорты + заглушки). Скилл `openspec-task-explain` будет проверять существование файлов из design.md — поэтому файлы должны существовать, но реализация не обязана быть полной.

## Что должно быть в `expected-output/`

Эталонный briefing, который скилл должен сгенерировать для одной из задач из tasks.md:

```
expected-output/
└── task-1.1.md           ← ожидаемый briefing для задачи 1.1
```

Этот briefing составляется **вручную**, по той же спеке `templates/briefing.md` из скилла, но без участия AI. Это эталон, с которым сравниваем результат one-shot запуска.

## Как использовать пример

1. Скопировать `input/` куда-то в рабочую директорию
2. Установить туда сгенерированный скилл: `gigacode extensions install ../skills/openspec-task-explain`
3. Запустить в GigaCode CLI: `выполни скилл openspec-task-explain для задачи 1.1`
4. Сравнить полученный `openspec/changes/<name>/briefings/task-1.1.md` с `expected-output/task-1.1.md`

## Рекомендуемая тема примера

Что-то узкое и понятное, чтобы и change.md, и design.md влезли по объёму, и было видно фильтрацию по задаче. Варианты:

- "Добавить валидацию платёжных запросов по schema X"
- "Реализовать JWT-аутентификацию для эндпоинта /reports"
- "Добавить retry-логику для внешнего вызова к сервису скоринга"

Главное — чтобы в design.md было **3-5 классов/модулей**, **2-3 контракта**, **3-5 edge cases**, и в tasks.md было **6-10 задач**. Тогда фильтрация по задаче имеет смысл и виден её эффект.
