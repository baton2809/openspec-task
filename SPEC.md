# SPEC: openspec-task-explain

## 1. Контекст и место в пайплайне OpenSpec

OpenSpec Skills — это набор скиллов для AI-агента (GigaCode CLI / Qwen Code / Claude Code), реализующих жизненный цикл работы со спецификациями: `teach → new-spec → propose → design → implement → apply-change → archive-change`.

Между шагом `design` (создан `change.md` + `design.md` + `tasks.md`) и `implement` (последовательная реализация всех задач) существует разрыв: разработчик, открывший задачу `[ ] 2.1 Реализовать X`, не имеет агрегированного контекста по этой одной задаче. Он либо читает все три документа целиком и собирает контекст в голове, либо запускает `implement` сразу — и теряет контроль.

Скилл `openspec-task-explain` закрывает этот разрыв: для одной выбранной задачи из `tasks.md` он собирает фокусированный briefing на основе `change.md` и `design.md` и выдаёт его пользователю. **Скилл не пишет код и не модифицирует артефакты OpenSpec.**

## 2. Цели и анти-цели

### Goals

- Дать разработчику фокусированный контекст по одной задаче из `tasks.md` — без необходимости читать все артефакты OpenSpec целиком.
- Использовать только то, что уже зафиксировано в `change.md` и `design.md`. Никаких выдумок и догадок.
- Явно отмечать пробелы: если в спеке нет данных, необходимых для реализации задачи — об этом сообщается отдельным разделом, а не маскируется красивой формулировкой.
- Цитировать спеку, а не пересказывать. Это ключевая защита от "испорченного телефона".

### Non-Goals

- Скилл **не** реализует задачу и не пишет код. Для этого есть `openspec-implement`.
- Скилл **не** модифицирует `tasks.md`, не ставит чекбоксы, не обновляет статусы.
- Скилл **не** валидирует существующий код на соответствие спеке. Для этого нужен отдельный скилл (`openspec-verify`, не входит в эту спецификацию).
- Скилл **не** ходит в Jira / Confluence / SberTrack / другие внешние системы. Только локальная файловая система.

## 3. Что должен сгенерировать GigaCode по этой спецификации

```
skills/openspec-task-explain/
├── SKILL.md                  ← основной файл скилла
└── templates/
    └── briefing.md           ← шаблон выходного документа
```

**Никаких других файлов создавать не нужно.** Тесты, README репозитория, примеры — уже подготовлены вручную и лежат в `examples/`.

## 4. Формат файла `skills/openspec-task-explain/SKILL.md`

Файл должен иметь точно такую структуру (по образцу `openspec-design/SKILL.md`):

### 4.1. Frontmatter (первая строка файла)

Одной строкой, через запятую:

```
name: openspec-task-explain description: Перед началом работы над задачей из tasks.md собрать фокусированный briefing на основе change.md и design.md — что делать, где, какие контракты и edge cases применимы, какие тесты нужны. Используй когда разработчик готов взяться за конкретную задачу и хочет получить агрегированный контекст без чтения всех артефактов целиком. НЕ используй для реализации — для этого есть openspec-implement. license: MIT compatibility: Требуется openspec/ layout в проекте, доступ к bash/git, AskUserQuestion tool. metadata: author: openspec-task-explain version: 1.0
```

### 4.2. Секция Input

```markdown
**Input**: Опционально — ID задачи в формате `N.M` (например, `2.1`) или имя change. Если не указано, скилл показывает невыполненные задачи и предлагает выбор.
```

### 4.3. Секция Bundle-пути

Описание где скилл ищет свои шаблоны:

```markdown
**Bundle-пути:**
- Собственные: `templates/X` — относительно директории этого скилла.
- Если harness даёт строку `Skill directory: <abs>` в обёртке активации — используй её как корень. Например: `Read("<skill dir>/templates/briefing.md")`.
- Если строки нет — один раз `Glob("**/openspec-skills/skills/openspec-task-explain/templates/<file>")`, бери первый результат.
- Пусто → спроси пользователя путь установки, не ищи вручную.
```

### 4.4. Секция Bundled materials

```markdown
**Bundled materials** (относительно этого SKILL.md):
- `templates/briefing.md` — шаблон структуры выходного briefing-документа.
```

### 4.5. Секция AskUserQuestion fallback

```markdown
**AskUserQuestion fallback**: если инструмент недоступен в клиенте — задай тот же вопрос обычным текстом и дождись ответа пользователя. Не угадывай.
```

### 4.6. Секция Steps

Девять пронумерованных шагов. Каждый шаг — короткое название и инструкция. Точное содержимое:

```markdown
## Steps

### 1. Определи change

Если имя change указано аргументом — используй его. Иначе:

```
ls openspec/changes/ 2>/dev/null
```

- Если ровно один active change (исключая `archive/`) — используй его автоматически.
- Если несколько — предложи пользователю выбор через AskUserQuestion.
- Если ни одного — сообщи "В проекте нет активных changes. Запусти `openspec-propose` для создания." и заверши.

Объяви: "Анализирую change: <name>".

### 2. Проверь наличие артефактов

Прочитай:
- `openspec/changes/<name>/change.md`
- `openspec/changes/<name>/design.md`
- `openspec/changes/<name>/tasks.md`

Если `design.md` отсутствует — сообщи: "Для этого change ещё не создан design.md. Запусти `openspec-design`." и заверши.
Если `tasks.md` отсутствует — сообщи: "Для этого change ещё не создан tasks.md. Запусти `openspec-design`." и заверши.
Если `change.md` отсутствует — сообщи: "Не найден change.md. Запусти `openspec-propose`." и заверши.

### 3. Определи задачу

Спарси `tasks.md` и собери список задач формата `[ ] N.M Описание` (только невыполненные, со статусом `[ ]`).

- Если ID задачи указан аргументом — найди задачу с этим ID. Если не находится — сообщи об этом и предложи список доступных. Не угадывай.
- Если ID не указан и невыполненных задач нет — сообщи "Все задачи в tasks.md отмечены как выполненные." и заверши.
- Если ID не указан и есть одна невыполненная задача — используй её, объяви выбор.
- Если несколько — через AskUserQuestion предложи список (максимум 10 пунктов; если задач больше — сообщи об этом и попроси указать ID).

Объяви: "Готовлю briefing по задаче N.M: <текст задачи>".

### 4. Прочитай шаблон briefing

Прочитай шаблон `templates/briefing.md` по правилам из секции Bundle-пути. Все плейсхолдеры в нём — формата `{{section_name}}`.

### 5. Собери секцию "Контекст из design.md"

Найди в `design.md` разделы, упоминающие задачу N.M или сущности из её формулировки. Извлеки:
- Соответствующий пункт из раздела "Изменения в коде" (если есть прямое упоминание задачи или её сущностей).
- Связанные Decisions из раздела "Decisions" — те, что относятся к области задачи.
- Goals/Non-Goals — если задача попадает в одну из них, укажи какую.

**Цитируй дословно**, обрамляя цитаты в `> ` (markdown blockquote). Под каждой цитатой указывай источник в формате `— design.md, раздел "<имя раздела>"`.

Если не находишь связи — пиши явно: "В design.md не найдено прямых упоминаний задачи N.M или её сущностей. Возможные причины: (1) задача описана только на уровне tasks.md без отдельного раздела в design.md, (2) задача относится к Non-Goals." Это ОК, не выдумывай связь.

### 6. Собери секцию "Целевые точки в коде"

Из раздела `Архитектура` → `Изменения в коде` (подразделы "Новые модули / классы / функции" и "Изменения в существующих сущностях") `design.md` извлеки таблицы и **отфильтруй строки**, относящиеся к задаче.

Критерий фильтрации: упоминание сущности в формулировке задачи ИЛИ упоминание задачи в столбце "Назначение/Изменение".

Для каждой целевой точки проверь, существует ли уже файл по указанному пути:

```
ls -la <Расположение>/<Имя> 2>/dev/null
```

- Если существует — пометь "(существует)".
- Если нет — пометь "(будет создан)".

Если в design.md нет ни одной строки, релевантной задаче — пиши: "В design.md не специфицированы целевые точки в коде для этой задачи." и переходи к секции 9 (Open Questions) — добавь туда вопрос: "Какие именно файлы должны быть созданы/изменены при реализации задачи N.M?"

### 7. Собери секции "Контракт", "Edge cases", "НФТ"

Из `design.md`:
- "Контракт" — раздел `Интерфейсы / Контракты`, отфильтрованный по сущностям задачи (входы/выходы/ошибки только релевантных контрактов).
- "Edge cases" — раздел `Обработка edge cases`, отфильтрованный по тем же сущностям.
- "Конфигурация" — упомяни только если задача затрагивает yml/properties параметры (раздел `Конфигурация → Изменения в yml/properties`).

Из `change.md` (раздел 9 "Хедеры и метаданные", раздел 10 "Валидация", раздел 11 "Безопасность", раздел 13 "Логирование", раздел 14 "Мониторинг"):
- "НФТ" — собери пункты из этих разделов, упоминающие сущности задачи. Если упоминаний нет, секцию выводи с текстом "Специфичных НФТ для задачи в change.md не указано — применяются общепроектные требования."

Все пункты в этих секциях — **цитаты с источниками**. Никаких пересказов.

### 8. Собери секцию "Чеклист тестов"

Из `design.md` раздел `Тестирование`:
- "Unit-тесты" — отфильтруй строки таблицы по задаче.
- "Integration-тесты" — отфильтруй аналогично.

Дополнительно из `tasks.md`, раздел "Чеклист верификации": найди подходящие пункты по основному / альтернативным / ошибочным сценариям, относящиеся к задаче.

Если ничего не находится — пиши явно: "В спеке не указаны конкретные тесты для задачи N.M. См. Open Questions." и добавь соответствующий вопрос в раздел 9.

### 9. Собери секции "Зависимости", "Риски", "Open Questions"

- "Зависимости от других задач" — пробеги по `tasks.md`, найди задачи с меньшими ID (`1.x`, или `N.M-1`), которые **могут быть пререквизитом**. Эвристика: если задача упоминает сущность, создаваемую другой задачей. Если эвристика срабатывает с уверенностью — указывай. Если нет — оставь раздел пустым с пометкой "Явные зависимости в tasks.md не специфицированы."
- "Риски и подводные камни" — раздел `Risks / Trade-offs` из `design.md`, отфильтрованный по сущностям задачи. Цитируй с источниками.
- "Open Questions" — раздел `Open Questions` из `design.md` со статусом "Открыт", + любые вопросы, накопленные на шагах 5-8.

### 10. Сформируй финальный briefing

Подставь собранные данные в шаблон `templates/briefing.md`. Сохрани результат в:

```
openspec/changes/<name>/briefings/task-N.M.md
```

Если директория `briefings/` не существует — создай её через `mkdir -p`.

Объяви пользователю: "Briefing готов: openspec/changes/<name>/briefings/task-N.M.md" и выведи его содержимое в чат.
```

### 4.7. Секция Output

Покажи пользователю **краткое резюме** в чате после сохранения файла, в формате:

```markdown
## Output

```

### Готовность к реализации

- ✓ Контракт определён
- ✓ Edge cases специфицированы
- ⚠ Тесты частично описаны (см. Open Questions)
- ✗ Целевые точки в коде не указаны (см. Open Questions)

### Рекомендация

Реализация ВОЗМОЖНА после ответа на N открытых вопросов.
ИЛИ: Реализация требует доработки design.md.
ИЛИ: Реализация готова к запуску `openspec-implement`.

### Next Steps

- Прочитай briefing полностью
- Ответь на Open Questions (если есть) и обнови design.md
- Запусти `openspec-implement` или работай руками по briefing
```
```

Маркеры готовности (`✓`, `⚠`, `✗`) проставляй автоматически:
- ✓ — секция собрана из спеки полностью.
- ⚠ — секция собрана частично, есть Open Questions.
- ✗ — секция пуста, информация в спеке отсутствует.

Финальная "Рекомендация" — выбери ровно одну из трёх в зависимости от количества `✗`:
- 0 крестиков → "Реализация готова к запуску `openspec-implement`."
- 1-2 крестика → "Реализация ВОЗМОЖНА после ответа на N открытых вопросов."
- 3+ крестика → "Реализация требует доработки design.md."

### 4.8. Секция Guardrails

Список из жёстких правил в стиле "ВСЕГДА / НЕ":

```markdown
## Guardrails

- ВСЕГДА цитируй спеку дословно через `> ` blockquote, под каждой цитатой указывай источник
- ВСЕГДА указывай статус каждой секции (✓/⚠/✗) — не маскируй пробелы красивыми формулировками
- НЕ пиши код и НЕ создавай файлы кроме `openspec/changes/<name>/briefings/task-N.M.md`
- НЕ модифицируй `tasks.md`, `design.md`, `change.md` — скилл строго read-only по отношению к ним
- НЕ ставь чекбоксы `[x]` — это работа `openspec-implement`
- НЕ придумывай edge cases которых нет в спеке — добавляй их в Open Questions
- НЕ угадывай ID задачи — если указанный ID не находится, спроси пользователя
- НЕ ходи в Jira/Confluence/внешние системы — только локальные файлы
- ЕСЛИ design.md или tasks.md отсутствуют — заверши работу с ясным сообщением, не пытайся работать частично
- ЕСЛИ задача упоминает классы/модули которых нет в коде — отметь это явно как "(будет создан)", не молчи
```

## 6. Контракт скилла (формальный)

**Вход:**
- Опциональный аргумент: ID задачи (`N.M`) или имя change
- Файлы: `openspec/changes/<name>/{change.md, design.md, tasks.md}`
- Файлы кода проекта (read-only, для проверки существования путей)

**Выход:**
- Создаётся файл `openspec/changes/<name>/briefings/task-N.M.md` по шаблону `briefing.md`
- В чат пользователя выводится Output-блок (см. 4.7)

**Ошибки и обработка:**
| Ситуация | Поведение |
|---|---|
| Нет активных changes | Сообщить, направить на `openspec-propose`, завершить |
| Нет design.md | Сообщить, направить на `openspec-design`, завершить |
| Нет tasks.md | Сообщить, направить на `openspec-design`, завершить |
| Указан ID, которого нет | Сообщить, показать список доступных, спросить уточнения |
| Все задачи выполнены | Сообщить и завершить |
| Несколько changes без аргумента | AskUserQuestion с выбором |
| Несколько задач без аргумента | AskUserQuestion с выбором (макс. 10) |

## 7. Edge cases

| Ситуация | Ожидаемое поведение |
|---|---|
| `tasks.md` пустой или малформед | Сообщить "tasks.md не содержит задач в формате `[ ] N.M ...`" и завершить |
| Задача в `tasks.md` есть, но без формата `N.M` (просто bullet) | Игнорировать, не предлагать в выборе |
| Несколько задач имеют одинаковый ID | Сообщить пользователю как баг tasks.md, попросить уточнить какую брать |
| `design.md` есть, но раздел "Архитектура" пустой | Все секции 2-5 помечаются `✗`, добавляются Open Questions, рекомендация → "доработка design.md" |
| `briefings/` уже содержит файл `task-N.M.md` | Перезаписать без вопроса (повторный запуск — нормальная ситуация) |
| Путь к коду из design.md содержит wildcards (например, `src/**/*.py`) | Не пытаться разрешить wildcards. Указать как есть, пометить "(шаблон пути)" |
| `change.md` использует не русские заголовки секций | Скилл должен искать секции по семантике (1-16 разделы), не по точному совпадению имени, но опираться на стандартный формат change.md из openspec-propose |

## 8. Что должно быть протестировано

**Скилл сам по себе тестов не требует** (это markdown + инструкции для агента). Тестирование — через прогон примера:

1. Положить пример в `examples/input/` (Python-проект с openspec-структурой и подготовленными `change.md`, `design.md`, `tasks.md`)
2. Запустить GigaCode CLI с командой "выполни скилл openspec-task-explain для задачи 1.1"
3. Сравнить полученный briefing с эталонным из `examples/expected-output/task-1.1.md`

Критерии успеха:
- Все 9 секций присутствуют
- Цитаты из design.md/change.md воспроизведены дословно
- Маркеры готовности (✓/⚠/✗) расставлены корректно
- Ни одной выдуманной цитаты или несуществующего раздела

## 9. Стиль и язык

- Skill.md и шаблон — на русском языке (как у остальных OpenSpec скиллов)
- Markdown-форматирование — точно как у `openspec-design/SKILL.md`: блок-цитаты через `>`, кодовые блоки через ```` ``` ````, заголовки через `##`/`###`
- Никаких эмодзи (кроме маркеров ✓/⚠/✗ и стандартных OpenSpec-маркеров вроде ⚠️ если они уже есть в шаблонах)
- Никаких HTML-комментариев в финальных файлах

## 10. Что не должно появиться в результате

- Любые файлы кроме `skills/openspec-task-explain/SKILL.md` и `skills/openspec-task-explain/templates/briefing.md`
- Никаких README, INSTALL, package.json, requirements.txt, .gitignore — этого делать НЕ нужно
- Никаких изменений в существующих файлах репозитория

## 11. Антипаттерны (что AI часто делает не так)

- Перефразирование цитат "своими словами" вместо дословного воспроизведения
- Добавление "полезных" разделов, которых нет в этой спеке (например, "Примерный код реализации")
- Самостоятельное определение релевантности через семантический поиск по всему design.md вместо явных правил из секции Steps
- Заполнение пустых секций "разумными предположениями" вместо явной отметки `✗` и Open Question
- Создание дополнительных файлов "для удобства" (README скилла, CHANGELOG, и т.п.)

Все эти антипаттерны — нарушение спеки, даже если результат "выглядит лучше".
